## 1. 背景与目标

用户管理接口是用户侧最基础的身份与资料管理能力，负责注册、登录、查看个人信息以及维护自己的账户状态。本阶段需要补齐 user 端自助管理接口，形成与 admin 侧一致的分层方式，便于后续统一接入 JWT、权限和审计逻辑。

---

## 2. 范围与边界

### 2.1 本次范围

- 开发 user 端用户管理接口（Controller + Service + Mapper + DTO/VO）。
- 支持注册、登录、当前用户信息查询、资料更新和账户软删除。
- 注册密码使用 BCrypt 加密存储。
- 所有查询和修改类操作都只作用于当前登录用户。

### 2.2 非本次范围

- admin 端用户管理能力。
- 找回密码、邮箱验证码、手机号绑定等扩展能力。
- 角色新增/切换/授权等后台能力。

## 3. 现状基线

### 3.1 数据库结构

#### `sys_user` 表

- `id`：主键，自增 bigint。
- `username`：登录用户名。
- `password`：BCrypt 密文。
- `nickname`：用户昵称。
- `email`：邮箱地址。
- `role_id`：关联 `sys_role`。
- `status`：账号状态（正常/封禁/软删除）。
- `create_time` / `update_time`：创建与更新时间。

#### `sys_role` 表

- 为用户提供默认角色和权限边界。
- 登录后根据角色决定可访问的用户能力范围。

### 3.2 认证与访问约束

- 注册和登录接口无需 JWT。
- 其余接口都必须携带 JWT。
- 修改、删除类接口必须将操作范围限制在当前 `userId`。

## 4. 接口设计

统一前缀：`/api/user`

### 4.1 用户注册

- 方法与路径：`POST /api/user/register`
- 请求体：

```json
{
	"username": "normal_user",
	"password": "123456",
	"confirmPassword": "123456"
}
```

- 处理逻辑：
	1. 校验用户名、密码、确认密码不能为空。
	2. 校验两次密码输入一致。
	3. 检查用户名是否唯一。
	4. 密码使用 BCrypt 加密后写入 `sys_user`。
	5. 默认角色、默认状态和默认昵称按系统约定初始化。

### 4.2 用户登录

- 方法与路径：`POST /api/user/login`
- 请求体：

```json
{
	"username": "zhangsan",
	"password": "123456"
}
```

- 处理逻辑：
	1. 校验用户名和密码不能为空。
	2. 查询用户是否存在。
	3. 校验账号状态是否正常。
	4. 比对 BCrypt 密码。
	5. 登录成功后返回 JWT。

### 4.3 获取当前用户信息

- 方法与路径：`GET /api/user/me`
- 处理逻辑：
	1. 从 JWT 中解析当前 `userId`。
	2. 查询当前用户的基础信息。
	3. 返回用户详情，密码字段不回传。

### 4.4 更新用户信息

- 方法与路径：`PUT /api/user/me`
- 请求体建议：

```json
{
	"nickname": "新的昵称",
	"email": "user@test.com"
}
```

- 处理逻辑：
	1. 仅允许修改当前登录用户自身的信息。
	2. 昵称、邮箱等资料字段按需更新。
	3. 存储空间配额等受限字段仅在具备对应权限时允许修改。

### 4.5 删除账户

- 方法与路径：`DELETE /api/user/me`
- 处理逻辑：
	1. 从 JWT 中解析当前 `userId`。
	2. 将账户状态更新为软删除状态。
	3. 保留历史数据，避免直接物理删除造成关联记录丢失。

## 5. 开发草案细化

### 5.1 统一返回与校验规则

- 登录和注册接口返回统一结果结构。
- 受保护接口统一依赖 JWT 鉴权。
- 所有密码处理都必须使用 BCrypt。

### 5.2 请求字段建议

#### 5.2.1 注册

- `username`：必填。
- `password`：必填。
- `confirmPassword`：必填。

#### 5.2.2 登录

- `username`：必填。
- `password`：必填。

#### 5.2.3 更新当前用户信息

- `nickname`：可选。
- `email`：可选。
- `maxStorageBytes`：仅在具备权限时允许修改。

### 5.3 返回结构建议

- 注册：返回成功状态，不强制返回用户详情。
- 登录：返回 JWT 字符串。
- 获取当前用户信息：返回 `UserDetailVO`。
- 更新/删除：返回统一成功状态即可。
