## 1. 背景与目标

笔记是用户端的核心内容单元，需要支持 Markdown 文件上传、内容解析、列表筛选、详情查看、内容更新、删除和全文搜索。本阶段先将 user 端笔记接口的文档结构统一到 admin 侧的写法，后续便于继续补充解析、渲染和缓存逻辑。

## 2. 范围与边界

### 2.1 本次范围

- 开发 user 端笔记接口（Controller + Service + Mapper + DTO/VO）。
- 仅支持通过上传 `.md` 文件创建或更新笔记内容。
- 支持按主题、标签等条件查询笔记列表。
- 支持返回笔记原文及渲染后的 HTML。
- 支持基于标题和内容的关键词搜索。

### 2.2 非本次范围

- admin 端笔记审核、发布和后台管理能力。
- 非 Markdown 格式文件的直接创建或更新。
- 图片压缩、裁剪等图像处理能力。

## 3. 现状基线

### 3.1 数据库结构

#### `biz_note` 表

- `id`：主键，自增 bigint。
- `user_id`：笔记所有者。
- `topic_id`：所属主题。
- `title`：笔记标题。
- `file_size`：笔记文件大小（字节）。
- `is_deleted`：软删除标记。
- `create_time` / `update_time`：创建与更新时间。

#### `biz_note_context` 表

- 存储笔记 Markdown 原文及相关上下文数据。

#### `biz_note_converted` 表

- 存储笔记渲染后的 HTML 结果以及转换缓存数据。

### 3.2 认证与访问约束

- 所有接口均需要 JWT 认证。
- 只有所有者可以访问、修改或删除自己的笔记。
- 上传时必须校验文件后缀为 `.md`。
- 上传和更新时都要校验文件大小是否在用户剩余存储配额内。

## 4. 接口设计

统一前缀：`/api/notes`

### 4.1 创建笔记

- 方法与路径：`POST /api/notes`
- 请求类型：`multipart/form-data`
- 请求参数：`.md` 文件。

```json
{
	"file": "multipart file"
}
```

- 处理逻辑：
	1. 校验文件存在且后缀为 `.md`。
	2. 校验文件大小是否在剩余存储配额内。
	3. 读取 Markdown 原文并解析笔记内容。
	4. 写入笔记主表和内容表。
	5. 返回创建成功结果。

### 4.2 查询笔记列表

- 方法与路径：`GET /api/notes`
- 处理逻辑：
	1. 仅查询当前用户的笔记。
	2. 支持按主题、标签等条件筛选。
	3. 返回分页或列表结果，按当前实现策略执行。

### 4.3 查看笔记详情

- 方法与路径：`GET /api/notes/{id}`
- 处理逻辑：
	1. 根据笔记 ID 查询详情。
	2. 校验笔记是否属于当前用户。
	3. 返回笔记原文和渲染后的 HTML。

### 4.4 更新笔记内容

- 方法与路径：`PUT /api/notes/{id}`
- 请求类型：`multipart/form-data`
- 请求参数：重新上传的 `.md` 文件。

- 处理逻辑：
	1. 校验笔记归属权。
	2. 校验文件后缀和文件大小。
	3. 用新上传的 Markdown 内容替换原有内容。
	4. 更新解析结果和渲染缓存。

### 4.5 删除笔记

- 方法与路径：`DELETE /api/notes/{id}`
- 处理逻辑：
	1. 校验笔记归属权。
	2. 执行软删除。
	3. 同步更新相关缓存或关联数据。

### 4.6 全文搜索

- 方法与路径：`GET /api/notes/search`
- 处理逻辑：
	1. 仅搜索当前用户的笔记。
	2. 基于标题和内容做关键词检索。
	3. 返回搜索结果列表。

## 5. 开发草案细化

### 5.1 上传约束

- 创建和更新接口都必须使用 `multipart/form-data`。
- 只允许 `.md` 文件进入业务层。
- 文件大小超过用户剩余配额时直接拒绝请求。

### 5.2 返回结构建议

- 创建笔记：返回成功状态。
- 列表查询：返回 `PageResult<NoteListVO>`。
- 详情查询：返回 `NoteDetailVO`。
- 更新和删除：返回统一成功状态即可。

### 5.3 访问控制

- 所有修改和删除接口都基于当前登录用户进行范围限制。
- 不允许访问其他用户的笔记详情或搜索结果。
