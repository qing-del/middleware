## 1. 背景与目标

图片是用户内容生产中的基础资源，主要用于笔记、主题和后续扩展场景。本阶段需要补齐 user 端图片接口，让用户能够上传图片、查看图片访问地址并删除自己的图片资源，同时保持与 admin 侧一致的接口组织方式。

## 2. 范围与边界

### 2.1 本次范围

- 开发 user 端图片接口（Controller + Service + Mapper + DTO/VO）。
- 支持图片上传、详情查询和删除。
- 上传时写入云存储并记录元数据。
- 删除时同时清理数据库记录和云端对象。
- 基于当前用户的剩余存储配额做大小校验。

### 2.2 非本次范围

## 3. 现状基线

### 3.1 数据库结构

#### `biz_image` 表

- `id`：主键，自增 bigint。
- `user_id`：图片所有者。
- `filename`：图片文件名。
- `oss_url`：图片访问 URL。
- `file_size`：图片大小（字节）。
- `create_time` / `update_time`：创建与更新时间。

### 3.2 认证与访问约束

- 所有接口均需要 JWT 认证。
- 只能操作当前登录用户自己的图片。
- 上传前必须校验文件大小是否超过用户剩余配额。

## 4. 接口设计

统一前缀：`/api/images`

### 4.1 上传图片

- 方法与路径：`POST /api/images`
- 请求类型：`multipart/form-data`
- 请求参数：图片文件。

```json
{
	"file": "multipart file"
}
```

- 处理逻辑：
	1. 校验请求是否携带图片文件。
	2. 校验文件大小是否在用户剩余配额内。
	3. 通过当前登录用户的 `userId` 生成并写入云存储。
	4. 保存图片元数据到 `biz_image`。
	5. 返回上传成功结果。

### 4.2 获取图片详情

- 方法与路径：`GET /api/images/{id}`
- 处理逻辑：
	1. 根据图片 ID 查询图片记录。
	2. 校验图片是否属于当前用户。
	3. 返回图片访问 URL 和元数据信息。

### 4.3 删除图片

- 方法与路径：`DELETE /api/images/{id}`
- 处理逻辑：
	1. 根据图片 ID 查询图片记录。
	2. 校验图片是否属于当前用户。
	3. 从云存储删除对应对象文件。
	4. 从数据库删除或软删除图片记录，按当前实现策略执行。

## 5. 开发草案细化

### 5.1 上传约束

- 上传请求必须使用 `multipart/form-data`。
- 文件大小校验应优先于持久化写入。
- 当前用户的剩余存储配额不足时应直接拒绝上传。

### 5.2 返回结构建议

- 上传图片：返回成功状态，必要时可回传图片 ID。
- 获取图片详情：返回 `ImageDetailVO`。
- 删除图片：返回统一成功状态。

### 5.3 权限约束

- 所有查询和删除操作都要基于当前登录用户的 `userId` 限定范围。
- 不允许用户访问或删除其他用户的图片。
