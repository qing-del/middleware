## 1. 背景与目标

主题用于承载用户自己的笔记分类，是列表筛选、公开控制和内容组织的基础能力。本阶段需要补齐 user 端主题接口，让用户可以维护自己的主题信息，并且在主题设为公开时强制校验审核状态。

## 4. 接口设计

统一前缀：`/api/topics`

### 4.1 查询主题列表

- 方法与路径：`GET /api/topics`
- 处理逻辑：
	1. 查询当前用户拥有的主题列表。
	2. 只返回当前登录用户自己的主题。
	3. 按当前实现策略返回列表或分页结果。

### 4.2 创建主题

- 方法与路径：`POST /api/topics`
- 请求体示例：

```json
{
	"topic_name": "我的主题",
	"description": "可选描述",
	"isPublic": 0
}
```

- 处理逻辑：
	1. 主题所有者自动设为当前登录用户。
	2. 写入主题基础信息。
	3. 默认按系统约定初始化审核状态。

### 4.3 查看主题详情

- 方法与路径：`GET /api/topics/{id}`
- 处理逻辑：
	1. 根据主题 ID 查询详情。
	2. 校验主题是否属于当前用户。
	3. 返回主题详细信息。

### 4.4 更新主题

- 方法与路径：`PUT /api/topics/{id}`
- 请求体示例：

```json
{
	"topic_name": "我的主题",
	"description": "可选描述",
	"isPublic": 1
}
```

- 处理逻辑：
	1. 校验目标主题是否属于当前用户。
	2. 修改主题标题、描述等字段。
	3. 当 `isPublic = 1` 时，先校验该主题已通过审核后才允许保存。

### 4.5 删除主题

- 方法与路径：`DELETE /api/topics/{id}`
- 处理逻辑：
	1. 校验主题归属权。
	2. 执行软删除，将 `is_deleted` 置为 `1`。
	3. 保留历史引用，避免直接物理删除。

## 5. 开发草案细化

### 5.1 请求字段建议

- `topic_name`：必填。
- `description`：可选。
- `isPublic`：可选，默认 `0`。

### 5.2 返回结构建议

- 查询列表：返回 `TopicListVO` 或分页结果。
- 详情查询：返回 `TopicDetailVO`。
- 创建、更新、删除：返回统一成功状态即可。

### 5.3 访问控制

- 所有修改和删除接口都必须以当前登录用户为边界。
- 如果主题要公开，必须先通过审核状态校验。
