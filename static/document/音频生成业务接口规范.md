# **音频生成业务与接口规范**

**文档版本**: 1.2.0

**核心架构**: Java (API/生产者) \-\> MySQL (持久化兜底) \-\> Redis Streams (任务队列) \-\> Python FastAPI (消费者/音频引擎) \-\> Nginx (资源代理)

## **1\. 业务流程与改造说明**

### **1.1 核心改造点 (对比原 GUI 脚本)**

* **废弃多版本合并**: 移除了原 drill\_generator.py 中解析 sequence=speed:1.0, speed:1.5, noise:1.0... 并进行长音频拼接的逻辑。  
* **单点精准生成**: 现在每次生成任务**必须明确指定单一的 speed (倍速) 和 noiseType (背景音类型)**，同时支持传入 noiseFactor 控制背景音量。  
* **资源解耦**: 客户端只需提交一次生成请求，Python 引擎处理完成后，生成单一对应的 MP3 文件，并通过 Nginx 暴露下载链接。

## **2\. 核心参数字典 (字典表设计)**

为了前后端以及 Python 引擎的参数统一，背景音类型需采用标准英文枚举或标识符映射。参考 app\_gui.py 中的 NOISE\_PROFILES：

### **2.1 背景音类型 (Noise Type)**

| 标识符 (noiseType) | 前端展示名称 (中文) | FFmpeg 底层滤波参数 (Python端使用) | 备注 |
| :---- | :---- | :---- | :---- |
| PURE | 纯净 (无背景音) | (空) | 仅生成 TTS 原声 |
| WHITE\_NOISE | 白噪音 | anoisesrc=c=white:r=44100 | 提升专注度 |
| PINK\_NOISE | 粉色噪音 | anoisesrc=c=pink:r=44100 | 更柔和的噪音 |
| BROWN\_NOISE | 褐噪音 | anoisesrc=c=brown:r=44100 | 低频轰鸣感 |
| CAFE | 咖啡厅 | anoisesrc=c=pink:r=44100,lowpass=f=2000 | 模拟环境白噪 |
| AIRPORT | 飞机场 | anoisesrc=c=brown:r=44100,lowpass=f=500,tremolo=f=0.5:d=0.3 | 模拟候机厅轰鸣 |
| SUBWAY | 地铁车厢 | anoisesrc=c=brown:r=44100,lowpass=f=300,tremolo=f=4.0:d=0.8 | 强烈的低频晃动感 |

### **2.2 语速倍率 (Speed Multiplier)**

* **数据类型**: Float / Double  
* **合法值范围**: 0.5 \~ 3.0  
* **常用预设**: 1.0 (原速), 1.25 (微快), 1.5 (较快), 2.0 (极快)  
* *引擎层转换注*: Python 处理时需将倍率转换为 Edge-TTS 的 rate 参数 (如 1.5 对应 \+50%)。

### **2.3 噪音因子/背景音量 (Noise Factor)**

* **数据类型**: Float / Double  
* **合法值范围**: 0.0 \~ 2.0 (0代表无背景音，1.0为标准生成音量)  
* **默认值**: 0.5 (通常背景音不要盖过人声，设置在0.2\~0.5之间听感较好)  
* *引擎层转换注*: Python 端将在 FFmpeg 混音时，向背景音轨道附加 volume={noiseFactor} 滤镜。

## **3\. 任务队列结构设计**

架构采用 **MySQL 任务表兜底 \+ Redis Streams 派发**，确保重 IO 的生成任务可靠执行。

### **3.1 数据库结构: audio\_tasks 表**

当用户提交生成请求时，Java 端首先落库生成此记录：

| 字段名 | 数据类型 | 说明 |
| :---- | :---- | :---- |
| id | BIGINT | 任务唯一主键 |
| user\_id | BIGINT | 提交该任务的用户 ID |
| source\_text | TEXT | 需要转换为音频的原始文本 |
| speed | DECIMAL(3,2) | 用户指定的语速 (例如: 1.50) |
| noise\_type | VARCHAR(32) | 背景音枚举 (例如: 'WHITE\_NOISE') |
| noise\_factor | DECIMAL(3,2) | **新增：背景音量因子 (例如: 0.50)** |
| status | TINYINT | 状态: 0-待处理(PENDING), 1-处理中(PROCESSING), 2-成功(SUCCESS), \-1-失败(FAILED) |
| result\_url | VARCHAR(255) | 成功后 Nginx 反代的音频下载链接 (如 /audio/12345.mp3) |
| error\_msg | VARCHAR(500) | 失败时的错误信息 |
| create\_time | DATETIME | 任务创建时间 |
| update\_time | DATETIME | 状态最后更新时间 |
| completed\_date | VARCHAR(10) / DATE | 任务完成的精确日期 (例如: '2026-05-19')。仅用于区分日期，方便后续定时任务清理。 |

### **3.2 Redis Streams 消息体结构**

Java 插入数据库后，将精简后的任务信息推入 Redis Stream (stream:audio:tasks)。Python 端作为 Consumer Group 读取。

**Stream Payload 示例 (JSON化表示，实际为 Hash 键值对):**

{  
  "taskId": "10086",  
  "userId": "999",  
  "speed": "1.5",  
  "noiseType": "WHITE\_NOISE",  
  "noiseFactor": "0.3",  
  "text": "Hello, this is a test audio generation."  
}

*注：考虑到 Redis 单条消息体积限制，如果 text 过长（如超过 10KB），可仅在队列传递 taskId，让 Python 直接连接数据库查询 source\_text。*

## **4\. API 接口规范**

### **4.1 用户端：提交音频生成任务**

**接口地址**: /user/audio/generate **请求方式**: POST **请求参数 (Body \- JSON)**:

| 参数名称 | 数据类型 | 是否必须 | 说明 |
| :---- | :---- | :---- | :---- |
| text | String | true | 待生成的纯文本内容 |
| speed | Float | true | 播放倍速 (0.5 \- 3.0) |
| noiseType | String | true | 背景音标识符 (如 PURE, CAFE 等) |
| noiseFactor | Float | false | 噪音音量因子，建议默认 0.5 |

**响应参数**:

{  
  "code": 0,  
  "msg": "任务已提交队列",  
  "data": {  
    "taskId": 10086,  
    "status": 0  
  }  
}

### **4.2 用户端：轮询/查询任务状态**

**接口地址**: /user/audio/status/{taskId}

**请求方式**: GET

**响应参数**:

{  
  "code": 0,  
  "msg": "success",  
  "data": {  
    "taskId": 10086,  
    "status": 2, // 0:排队中, 1:合成中, 2:已完成, \-1:失败  
    "resultUrl": "\[http://127.0.0.1/audio/10086\_WHITE\_NOISE\_1.5x\_v0.3.mp3\](http://127.0.0.1/audio/10086\_WHITE\_NOISE\_1.5x\_v0.3.mp3)",  
    "errorMsg": ""  
  }  
}

### **4.3 Python 引擎回调 Java 接口 (双重 Webhook)**

为保证状态机完整与空间利用率，Python 引擎的执行周期内将发起两次回调通知 Java。

#### **4.3.1 回调 A：任务开始处理 (Picked Up)**

当 Python 消费者从 Redis 弹出任务，准备开始调用 Edge-TTS 和 FFmpeg 前调用此接口。

**接口地址**: /admin/audio/callback/start (内网调用)

**请求方式**: POST

**请求参数 (Body)**:

{  
  "taskId": 10086  
}

#### **4.3.2 回调 B：任务处理完成 (Completed/Failed)**

当 Python 引擎生成好本地 MP3 文件，或者在生成过程中捕获到异常时，调用此接口。

**接口地址**: /admin/audio/callback/finish (内网调用)

**请求方式**: POST

**请求参数 (Body)**:

{  
  "taskId": 10086,  
  "status": 2, // 2: 成功, \-1: 失败  
  "resultUrl": "/audio/10086\_WHITE\_NOISE\_1.5x.mp3",  
  "errorMsg": ""   
}

**响应参数 (非常关键)**:

Python 需要根据 Java 的响应决定是否保留本地文件：

{  
  "code": 0, // 0表示接口通讯成功  
  "msg": "success",  
  "data": true // true: Java端数据库状态更新成功，false: 状态更新失败  
}

**Python 端联动逻辑 (文件清理)**：如果 Python 收到 Java 响应 code \!= 0 或 data \== false，则说明该结果无需保留，**Python 端必须立刻执行文件删除逻辑（os.remove(...)），回收磁盘空间。** \---

## **5\. Python 引擎执行逻辑 (完整闭环流)**

Python 消费者从 Redis 拿到任务后的执行流：

1. **解析参数**: 获取 taskId, text, speed, noiseType, noiseFactor。  
2. **触发回调 A (Start)**: 调用 /admin/audio/callback/start 接口，告知 Java 开始处理该任务。  
3. **计算真实 Rate**: edge-tts \--rate=+50% (计算公式根据 speed 决定)。  
4. **调用 Edge-TTS**: 将纯净的人声文本生成为 temp\_vocal.mp3。  
5. **加载背景音与音量调节**:  
   * 根据 noiseType 获取 FFmpeg 字符串 (如 anoisesrc=c=white:r=44100)。若为 PURE，则跳过混音直接走第 7 步。  
   * **混音逻辑补充**: 在合成时，通过 FFmpeg 的 volume 滤镜挂载 noiseFactor 因子。例如通过类似 \[1:a\]volume=0.3\[bg\];\[0:a\]\[bg\]amix... 的 filter\_complex 将噪音轨道压低音量。  
6. **FFmpeg 混音 (Mix)**: 将 temp\_vocal.mp3 与经过调音的背景滤镜结合，输出最终文件 final\_{taskId}.mp3。  
7. **触发回调 B (Finish) 与状态检查**:  
   * 移动文件到 Nginx /data/drills/audio/ 目录。  
   * 调用 /admin/audio/callback/finish 接口提交结果。  
   * 检查响应值：如果回调失败（data: false），则 Python 尝试 os.remove 删掉生成的音频文件以节约空间。  
8. **ACK 确认**: 最后，无论业务成功或失败，向 Redis 发送 XACK 确认任务已被处理，移出 Pending 列表。
